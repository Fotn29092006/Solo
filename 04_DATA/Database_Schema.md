Current MVP spine tables:

- `profiles`: Internal app profile UUID, Telegram identity binding, display name, level, XP, and rank key.
- `goals`: selected user goals.
- `user_paths`: selected progression path.
- `daily_quests`: assigned daily quest records.
- `quest_completions`: completion records for daily quests.
- `weekly_checkins`: weekly weigh-in and reflection inputs.
- `xp_events`: append-style XP history.
- `streaks`: current and best streak counters.

Logging tables:

- `water_logs`: append-only hydration events owned by `profile_id`, with client event idempotency, amount in ml, UTC log date, recent-history indexes, and RLS enabled without anon/client policies.
- `workout_logs`: append-only session logs owned by `profile_id`, with client event idempotency, workout type, optional workout name, optional duration/RPE, UTC log date, recent-history indexes, and RLS enabled without anon/client policies.
- `sleep_logs`: append-only sleep duration logs owned by `profile_id`, with client event idempotency, duration in minutes, optional sleep quality, optional morning energy, UTC log date, recent-history indexes, and RLS enabled without anon/client policies.
- `meal_logs`: append-only meal logs owned by `profile_id`, with client event idempotency, meal type, optional meal name, optional calories/protein, UTC log date, recent-history indexes, and RLS enabled without anon/client policies.

Identity and ownership:

- `profiles.id` is the internal app profile ID generated by Postgres.
- `profiles.telegram_user_id` is the unique external Telegram identity binding.
- Telegram Mini App `initData` must be validated server-side before any profile lookup or write.
- Server routes must derive `profile_id` from the validated Telegram user ID.
- Clients must not send trusted `profile_id` values.
- `/api/home` seeds `daily_quests` for the current day only after deriving `profile_id` from validated Telegram identity.
- `/api/home` reads live streak records for the validated profile.
- `/api/home` reads the current server-computed weekly check-in for the validated profile.
- `/api/quests/complete` writes `quest_completions`, `xp_events`, updates `daily_quests.status`, syncs `profiles.total_xp`/`profiles.level`, and updates the `daily_quest` streak after deriving `profile_id` from validated Telegram identity.
- `/api/weekly-checkin` reads or upserts one `weekly_checkins` row for the current `profile_id + week_start_date`, generates the MVP summary server-side, and updates `weekly_review` streak after deriving `profile_id` from validated Telegram identity.
- `/api/water-logs` writes `water_logs` only after deriving `profile_id` from validated Telegram identity. The client may send only a `client_event_id` for idempotency, an `amount_ml`, and optional note; unknown fields are rejected.
- `/api/home` reads today's `water_logs` total for the validated profile and returns `waterToday`.
- `/api/workout-logs` writes `workout_logs` only after deriving `profile_id` from validated Telegram identity. The client may send only `client_event_id`, `workout_type`, optional `workout_name`, optional `duration_min`, optional `rpe`, and optional note; unknown fields are rejected.
- `/api/home` reads today's `workout_logs` session count and total duration for the validated profile and returns `workoutToday`.
- `/api/sleep-logs` writes `sleep_logs` only after deriving `profile_id` from validated Telegram identity. The client may send only `client_event_id`, `sleep_duration_min`, optional `sleep_quality`, optional `morning_energy`, and optional note; unknown fields are rejected.
- `/api/home` reads today's `sleep_logs` total duration for the validated profile and returns `sleepToday`; sleep aggregate read failures are soft and do not fail the full Home payload.
- `/api/meal-logs` writes `meal_logs` only after deriving `profile_id` from validated Telegram identity. The client may send only `client_event_id`, `meal_type`, optional `meal_name`, optional `calories`, optional `protein_g`, and optional note; unknown fields are rejected.
- `/api/home` reads today's `meal_logs` count for the validated profile and returns `mealToday`; meal aggregate read failures are soft and do not fail the full Home payload.

RLS status:

- The MVP migration enables RLS on all user-owned tables.
- The MVP migration intentionally does not create anon/client policies yet.
- Until Supabase Auth/JWT mapping is introduced, data writes should go through server routes using validated Telegram identity and privileged Supabase access.
- Every user-owned table must have a production RLS strategy before direct client reads or writes are enabled.

Deferred tables:

- Ingredient-level food logs.
- Notification dispatch logs.
- Achievements.
- Domain score derivations.
