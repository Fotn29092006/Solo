Implementation rules:

- Client uses anon key only.
- Server-side privileged actions use secure environment variables.
- Notification dispatch should use server-side code.
- The client must not send trusted `profile_id` values.
- Server routes must derive profile ownership from validated Telegram init data.
- RLS must remain enabled on user-owned tables.
- Direct client policies are deferred until a Supabase Auth/JWT mapping exists.

Identity model:

- `profiles.id` is an internal app profile UUID generated by Postgres.
- `profiles.telegram_user_id` is the unique external Telegram identity binding.
- The MVP server path validates Telegram Mini App init data, then looks up or creates a profile by `telegram_user_id`.
- Supabase Auth `auth.uid()` is not the current MVP identity source.

Apply migration:

- This migration is prepared for a fresh Supabase development target or clean migration history.

- Applying from this workspace requires a Supabase CLI login/link or a database connection path, not only anon/service-role API keys.
